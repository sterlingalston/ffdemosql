Database Overview

Purpose: This database represents a comprehensive financial technology infrastructure similar to Fiserv's platform, designed for educational and demonstration purposes. It supports core banking, payment processing, card services, digital banking, risk management, compliance, and analytics.
Important: This is educational demo data only and not based on actual Fiserv systems. All data is fictional and should not be used in production environments.

CORE Schema

The core schema contains fundamental entities that support all banking operations.

FINANCIAL_INSTITUTIONS

Purpose: Stores information about banks, credit unions, and other financial institutions that use the platform.
Field Name Data Type Constraints Description
INSTITUTION_ID INT IDENTITY Primary Key Unique identifier for each institution
INSTITUTION_CODE VARCHAR(20) NOT NULL, UNIQUE Short code identifier
INSTITUTION_NAME VARCHAR(200) NOT NULL Full institution name
ROUTING_NUMBER VARCHAR(9) NOT NULL, UNIQUE ABA routing number
SWIFT_CODE VARCHAR(11) NULL International SWIFT code
REGULATION_LEVEL VARCHAR(50) NOT NULL Regulatory authority (FDIC, NCUA, OCC)

CUSTOMERS

Purpose: Stores customer information for both personal and business accounts.
Field Name Data Type Constraints Description
CUSTOMER_ID INT IDENTITY Primary Key Unique customer identifier
INSTITUTION_ID INT Foreign Key References FINANCIAL_INSTITUTIONS
CUSTOMER_TYPE VARCHAR(20) NOT NULL Personal or Business
SSN_ENCRYPTED VARBINARY(256) NULL Encrypted Social Security Number
CREDIT_SCORE INT NULL Credit score (300-850)
KYC_STATUS VARCHAR(20) Default 'Pending' Know Your Customer status
Unique Constraint: (INSTITUTION_ID, CUSTOMER_NUMBER) - Ensures customer numbers are unique within each institution.

PRODUCT_TYPES

Purpose: Defines banking products and their characteristics.
Field Name Data Type Description
PRODUCT_CODE VARCHAR(20) Unique product identifier
PRODUCT_CATEGORY VARCHAR(50) Checking, Savings, Certificate, Credit, Business
INTEREST_RATE DECIMAL(5,4) Annual percentage rate
MINIMUM_BALANCE DECIMAL(12,2) Required minimum balance

ACCOUNTS

Purpose: Customer account instances with current balances and status.
Key Relationships:
  • Each account belongs to one customer
  • Each account is of one product type
  • Customers can have multiple accounts

PAYMENTS Schema

Manages payment processing networks and transaction handling.

PAYMENT_NETWORKS

Purpose: Defines available payment networks and their characteristics.
Network Type Settlement Time Typical Use Case
ACH 1-3 Business Days Payroll, bill pay, transfers
Wire Transfer Same Day Large transactions, real estate
RTP Real Time Instant payments, emergencies
P2P Transfer Real Time Person-to-person payments

TRANSACTIONS

Purpose: Records all payment transactions across the platform.
Business Rules:
  • Either DEBIT_ACCOUNT_ID or CREDIT_ACCOUNT_ID must be internal to the institution
  • External accounts require routing number and bank name
  • All transactions require risk scoring
  • Status progression: Pending → Processing → Completed/Failed

CARDS Schema

Comprehensive card services including debit and credit card management.

CARD_PROGRAMS

Purpose: Defines card program templates with features and fees.
Card Type Network Features
Debit Visa/MasterCard Direct account access, no credit limit
Credit Visa/MasterCard/Amex Credit line, rewards, interest charges

CARD_ACCOUNTS

Purpose: Individual card instances issued to customers.
Security Note: Card numbers and CVV codes are stored encrypted. Only masked versions are used for display purposes.

CARD_TRANSACTIONS

Purpose: Records all card-based transactions including purchases, payments, and cash advances.
Transaction Flow:
  1. Authorization received from network
  2. Risk scoring and fraud detection
  3. Transaction posting to account
  4. Rewards calculation (if applicable)

DIGITAL_USERS

Security Feature Implementation

Two-Factor Auth SMS or email verification

USER_SESSIONS

Purpose: Tracks active user sessions for security and analytics.

BILL_PAY_PAYEES

Purpose: Customer-defined payees for bill payment services.

RISK Schema

Risk management and fraud detection systems.

RISK_RULES

Purpose: Configurable rules for detecting suspicious activity.
Rule Type Description Typical Threshold
Transaction Large or unusual amounts $10,000+
Velocity Too many transactions quickly 5+ in 1 hour
Location Unusual geographic patterns 500+ miles from normal
Authentication Failed login attempts 3+ failures

RISK_EVENTS

Purpose: Records triggered risk events for investigation and resolution.
Event Lifecycle: Open → Under Review → Resolved/Escalated

COMPLIANCE Schema

Regulatory compliance and reporting infrastructure.

REGULATORY_REPORTS

Purpose: Tracks regulatory report generation and submission.
Report Type Regulator Frequency
Call Report FDIC Quarterly
NCUA 5300 NCUA Quarterly
BSA/AML Report FinCEN As needed
CRA Report FDIC/OCC Annual

SANCTIONS_SCREENING

Purpose: OFAC and sanctions list screening for compliance.
Screening Process: All customers and transactions are automatically screened against OFAC SDN, Non-SDN, and other sanctions lists. Match scores above thresholds require manual review.

ANALYTICS Schema

Business intelligence and performance metrics.

DAILY_METRICS

Purpose: Daily aggregated metrics for business intelligence and monitoring.
Metric Category Key Indicators
Transaction Volume Count, dollar volume, average size
Customer Growth Total customers, new acquisitions
Digital Adoption Active digital users, engagement
Risk Management Fraud detection, loss amounts
System Performance Uptime, compliance alerts
Data Aggregation: Metrics are calculated nightly from operational data and stored for fast reporting and dashboard generation.
